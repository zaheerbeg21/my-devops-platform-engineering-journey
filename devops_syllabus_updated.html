INTERVIEW QUESTION
A process on your server is consuming 100% CPU. Walk me through how you diagnose and resolve it.
Answer framework: 1) Use top or htop to identify the PID. 2) Use ps aux --sort=-%cpu | head -10 to see top consumers. 3) Use strace -p [PID] to see syscalls in real time. 4) Check /proc/[PID]/ for memory maps, open files. 5) If it's Odoo — check for runaway ORM query via pg_stat_activity. 6) Decision: kill, renice, or restart service. 7) Root cause via application logs. Always log the incident.

Enterprise Use Case — Must Build This

Odoo Database Backup Automation

Write a production-grade Bash script that: dumps PostgreSQL Odoo database with timestamp, compresses using gzip, uploads to Azure Blob Storage using az-cli, retains last 30 daily backups and 12 monthly, sends Slack notification on success/failure, and logs all operations to /var/log/odoo-backup.log. This is a real deliverable you can show clients and interviewers.

#!/bin/bash set -euo pipefail BACKUP_DIR="/opt/backups/odoo" DATE=$(date +%Y%m%d_%H%M%S) DB_NAME="odoo_production" AZURE_CONTAINER="odoo-backups" pg_dump -U odoo -h localhost \ -F c $DB_NAME | gzip > \ $BACKUP_DIR/backup_$DATE.dump.gz az storage blob upload \ --container-name $AZURE_CONTAINER \ --file $BACKUP_DIR/backup_$DATE.dump.gz \ --name backup_$DATE.dump.gz # Retain last 30 backups only find $BACKUP_DIR -name "*.gz" \ -mtime +30 -delete
INTERVIEW QUESTION
How do you ensure a bash script is production-safe and won't silently fail?
Answer: Always start with set -euo pipefail. -e exits on error, -u treats unset variables as errors, -o pipefail fails if any command in a pipe fails. Add proper logging with timestamps. Use trap for cleanup on EXIT, INT, TERM signals. Validate all inputs and environment variables at the start. Never assume files exist — check first. Add idempotency so the script can be safely re-run. Test failure scenarios, not just happy paths.

Explain Trunk-based development vs Git Flow. When would you recommend each?
Trunk-based: All developers commit to a single main branch, using short-lived feature branches (max 1–2 days). Requires feature flags for incomplete work. Enables true CI/CD. Best for: teams deploying multiple times per day, mature CI pipelines, experienced teams. Git Flow: Long-lived develop, feature, release, hotfix branches. More structured. Best for: scheduled release cycles, teams with QA gates, less mature CI setups. Your recommendation for GCC enterprise clients: Git Flow for ERP (Odoo has long release cycles), Trunk-based for FastAPI microservices (frequent deployments).

INTERVIEW QUESTION
Your Odoo application is responding slowly. Users report 30-second load times. How do you diagnose this systematically?
Layered diagnosis: 1) DNS resolution — dig odoo.company.com. 2) Network latency — ping + traceroute. 3) Nginx — check access.log for response times, upstream_response_time. 4) Odoo app — check /var/log/odoo/odoo-server.log for slow methods. 5) PostgreSQL — run SELECT query, mean_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10. 6) System resources — vmstat, iostat, free -h. 7) Concurrent connections — check pg_stat_activity, Odoo worker count. Most likely culprits: missing DB index, Odoo worker count too low, PostgreSQL query without index.

INTERVIEW QUESTION
What is a multi-stage Docker build and why is it important for production?
Answer: Multi-stage builds use multiple FROM statements. Early stages compile/build the application (including dev dependencies, build tools, compilers). The final stage copies only the compiled artifacts from earlier stages into a minimal base image. Why it matters: 1) Dramatically smaller final images (Next.js: 1.2GB dev → 150MB production). 2) No build secrets, dev tools, or source code in production image. 3) Reduced attack surface. 4) Faster deployment pulls. Example for Next.js: Stage 1 (node:18) installs all deps and runs next build. Stage 2 (node:18-alpine) copies only .next/standalone and serves it. Result: ~85% image size reduction.

INTERVIEW QUESTION
How do you manage secrets in Docker Compose for production? What should you never do?
What NEVER to do: Never put secrets in docker-compose.yml directly, never in image build args (they appear in layer history), never in environment variables in the Dockerfile. Correct approaches for production: 1) Docker Secrets (Swarm mode) — secrets mounted as files at /run/secrets/. 2) External secrets manager — HashiCorp Vault, Azure Key Vault, AWS Secrets Manager. Application fetches at runtime, not build time. 3) Environment file injection at deploy time — .env file not in repo, injected by CI/CD pipeline. 4) Kubernetes Secrets (when using K8s). For Compose specifically: Use env_file: .env.prod where .env.prod is in .gitignore and managed by your deployment system.

INTERVIEW QUESTION
A security scan found a HIGH severity CVE in your Odoo base image. How do you handle this in a CI/CD pipeline?
Systematic response: 1) Scan early — integrate Trivy into the CI pipeline, run on every build before push. 2) Define policy — HIGH severity blocks the pipeline, CRITICAL triggers immediate incident. MEDIUM/LOW goes to backlog. 3) For the specific CVE: check if it affects your usage (many CVEs are in packages you don't use). 4) Fix path: rebuild with updated base image. 5) If upstream hasn't patched: consider multi-stage build to install patched version manually. 6) Add CVE to allowlist with expiry date and Jira ticket if a temporary bypass is needed. 7) Document all CVEs and their status in a security runbook. 8) Never deploy CRITICAL or unreviewed HIGH severity containers to production.

INTERVIEW QUESTION
Explain the difference between Blue-Green and Canary deployment. When would you use each?
Blue-Green: Two identical environments (Blue = current prod, Green = new version). You deploy to Green, test it, then switch load balancer from Blue to Green. Rollback is instant (switch back). Best for: stateless apps, high-risk deployments, when you can afford double the infrastructure cost. Canary: New version receives small % of traffic (5%), monitor errors/latency, gradually increase if healthy. Best for: high-traffic apps, when you want real user validation, stateful services. For your stack: Use Blue-Green for Odoo ERP releases (complex, high-risk, infrequent). Use Canary for FastAPI microservices (frequent deployments, easy to roll back at load balancer level).

INTERVIEW QUESTION
Your CI pipeline takes 45 minutes. How do you reduce it to under 10 minutes?
Systematic optimization: 1) Measure first — add timing to each step, identify the slowest (usually tests + Docker builds). 2) Parallelize — run lint, test, security scan as parallel jobs. 3) Cache aggressively — pip/npm packages, Docker layers, test results. 4) Split test suites — run unit tests in CI on every commit (fast), integration tests only on merge to main. 5) Optimize Docker builds — order layers from least to most frequently changing (COPY requirements.txt before COPY ./app). 6) Use matrix builds carefully — don't test Python 3.8/3.9/3.10 on every commit. 7) Use self-hosted runners for heavy jobs. 8) Implement test parallelization (pytest-xdist). Real-world result: FastAPI pipeline from 28min → 7min using these techniques.

INTERVIEW QUESTION
How do you handle Terraform state in a team environment, and what problems arise with state?
The state problem: Default local state breaks in teams — two engineers running apply simultaneously corrupt state. Solution — Remote state with locking: Store state in Azure Blob (or S3/Terraform Cloud). Backend config: backend "azurerm" { storage_account_name, container_name, key }. Azure Blob provides state locking via leases — second apply attempt waits until lock is released. State gotchas to know: 1) Never manually edit state file. 2) Use terraform state mv for refactoring, not delete+recreate. 3) terraform import for existing resources. 4) Sensitive values in state are stored in plaintext — encrypt your storage. 5) State per workspace (dev/staging/prod) isolates blast radius. State drift: Use terraform plan in CI on every PR to detect drift before it reaches production.

INTERVIEW QUESTION
Explain the difference between a Deployment and a StatefulSet. Why does Odoo require special consideration in Kubernetes?
Deployment: Manages stateless pods. Pods are interchangeable — K8s can kill any pod and replace it. Network identity changes with each pod restart. Best for: FastAPI, Next.js, any 12-factor app. StatefulSet: Manages stateful pods. Gives each pod a stable network identity (pod-0, pod-1) and stable storage. Ordered, graceful scaling. Best for: PostgreSQL, Elasticsearch, Zookeeper. Odoo complexity in K8s: Odoo stores attachments on the filesystem (/var/lib/odoo/filestore). In K8s, this must be backed by a PersistentVolume. But if you run 3 Odoo replicas, all 3 need access to the same filestore — requiring ReadWriteMany (RWX) PVC, typically Azure Files or NFS. This is expensive and slow. Better pattern: Migrate Odoo filestore to Azure Blob Storage (S3-compatible) using the ir.attachment model override. Then Odoo becomes stateless and can run as a normal Deployment.

INTERVIEW QUESTION

CAT-1 Docker & Containers (25 Questions) ▶
FOUNDATIONAL
What is the difference between a Docker image and a container?
Image: Read-only immutable blueprint. A layered filesystem snapshot. Like a class in OOP. Stored in registry. Container: Running instance of an image. Has a writable layer on top. Like an object instantiated from a class. Ephemeral by default — data is lost when container stops unless using volumes. Key insight for interviews: Never store application data inside a container's writable layer. Always use named volumes or bind mounts.
INTERMEDIATE
How do you handle database schema migrations in a containerized environment?
Problem: Schema migration must run exactly once before new app version starts, not on every pod startup. Solutions: 1) Init container (K8s) that runs migrations before app container starts. 2) Docker Compose: migration service with depends_on app. 3) Helm hook: pre-upgrade Job that runs Alembic/Django migrate before deployment. 4) Separate migration pipeline step in CI/CD. For FastAPI + Alembic: Kubernetes initContainer runs alembic upgrade head, only starts app container on exit code 0. Critical: Migrations must be backward-compatible if you're running rolling deployments (old + new versions run simultaneously briefly).
ADVANCED
How would you run 100 Docker containers on a single server efficiently?
Resource optimization: 1) Set memory limits on every container (without limits, one container can OOM the host). 2) Use CPU quotas (not reservations) — quotas limit burst, reservations guarantee. 3) Share base image layers — if all 100 containers use the same Python base, it's stored once. 4) Use alpine-based images to minimize filesystem overhead. 5) Limit container logging (json-file driver with max-size=50m). 6) Use container namespaces and cgroups properly. 7) Consider switching to Kubernetes for better bin-packing. 8) Use Docker's --memory-swap to control swap behavior. At 100 containers you likely need K8s or at minimum Docker Swarm for proper scheduling.
CAT-2 CI/CD & Pipelines (25 Questions) ▶
SCENARIO-BASED
A developer accidentally pushed a secret (API key) to GitHub. What do you do?
Immediate response (within minutes): 1) Revoke the compromised secret immediately at the source (AWS Console, Azure Portal, API provider). Don't wait to understand scope — revoke first. 2) Generate a new secret. 3) Update the secret in your secrets manager (Key Vault, Secrets Manager). 4) The secret is in Git history — it cannot be "deleted." Assume it's compromised. Git cleanup (after revocation): Use git filter-repo or BFG Repo Cleaner to remove from history, then force push (this rewrites history — coordinate with team). Prevention: Install GitLeaks as pre-commit hook. Add detect-secrets to CI. Use GHAS (GitHub Advanced Security) secret scanning. Set up automatic revocation via GitHub's secret scanning push protection.
SYSTEM DESIGN
Design a CI/CD pipeline for an ERP system (like Odoo) that requires minimal downtime during deployments.
Odoo-specific challenges: Stateful (filestore), long-running background jobs, DB migrations, multi-process workers. Strategy: 1) Pre-deploy: run DB migrations in maintenance mode (short window). 2) Rolling restart: use Odoo's --stop-after-init for worker management. 3) Health check gate: new instance must pass health check before old one is killed. 4) Session handling: Redis for session storage (not file-based) so sessions survive pod restart. 5) Zero-downtime if no schema changes: replace Odoo containers while keeping DB and Redis intact. 6) Major version upgrades (16→17): blue-green with DB snapshot, test migration, cutover. Key insight: You can't have true zero-downtime Odoo deployments with breaking DB migrations — honest communication with clients about maintenance windows is part of architect maturity.
CAT-3 Kubernetes (25 Questions) ▶
CONCEPTUAL
What is the difference between liveness, readiness, and startup probes?
Liveness probe: "Is this container alive?" If it fails, K8s restarts the container. Use for: detecting deadlocks, infinite loops, corrupted state that won't auto-recover. Don't make it too aggressive — false positives cause restart loops. Readiness probe: "Is this container ready to receive traffic?" If it fails, K8s removes pod from Service endpoints. Use for: slow startup, temporary inability to serve (DB disconnected, cache warming). This is how you achieve zero-downtime deployments. Startup probe: "Has the app started yet?" Disables liveness/readiness during startup period. Critical for slow-starting apps like Odoo (can take 30–60 seconds). Without it, liveness kills the container before it finishes starting. For Odoo: Startup probe: HTTP /web/health with failureThreshold=30 (allows 5 min startup). Readiness: same endpoint. Liveness: same endpoint but less frequent.
ADVANCED
How does Kubernetes handle node failure, and what do you need to configure for true HA?
What K8s does automatically: Node Controller detects unreachable node after ~5 minutes. Taints node with NoSchedule. Evicts pods. Reschedules on healthy nodes. What you must configure for true HA: 1) Minimum 3 nodes (1 can fail, quorum maintained). 2) Pod anti-affinity rules so replicas don't land on same node. 3) PodDisruptionBudget: ensure at least 1 pod always running during voluntary disruptions. 4) Multiple replicas per Deployment (not 1). 5) Resource requests set on all pods (required for scheduler to make placement decisions). 6) ReadinessProbes so unhealthy pods don't receive traffic. 7) Stateful components (PostgreSQL) need their own HA solution — don't rely on K8s to solve database HA. Use Azure Database for PostgreSQL Flexible Server with ZoneRedundant mode.
CAT-4 Terraform & IaC (20 Questions) ▶
PRACTICAL
Someone manually changed a resource in Azure that was created by Terraform. How do you detect and handle this?
Detection: terraform plan shows drift between desired state (in code) and real state (in Azure). The plan will show changes Terraform wants to make to bring resource back in line. Options: 1) Revert drift: run terraform apply — Terraform overwrites the manual change. 2) Accept drift: update Terraform code to match the manual change, then apply. 3) Investigate first: was the manual change intentional (emergency fix) or accidental? Prevention: Run terraform plan on a schedule in CI (e.g., nightly) and alert on drift. Use Azure Policy to deny changes outside Terraform. Implement strict RBAC: only the CI pipeline's service principal has write access. Document "break glass" procedure for emergency manual changes, with mandatory ticket. Key insight: Drift is a process problem, not just a tooling problem. Establish culture of "infrastructure changes go through code."
